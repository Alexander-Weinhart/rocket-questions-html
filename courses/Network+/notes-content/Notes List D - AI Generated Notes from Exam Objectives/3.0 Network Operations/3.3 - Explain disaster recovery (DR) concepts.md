# 3.3 - Disaster Recovery Concepts

> **Big idea:** Disaster recovery is about preparing for serious failure before it happens. The goal is to restore critical services within acceptable time and data-loss limits using backups, redundancy, alternate sites, tested procedures, and clear priorities.

## What you should be able to explain

- Explain RTO, RPO, MTTR, MTBF, and SLAs.
- Compare backup types and recovery site options.
- Understand redundancy, failover, and high availability.
- Recognize why testing matters.

## Concept map

| Term | Meaning | Question it answers |
| --- | --- | --- |
| RTO | Recovery time objective | How long can it be down? |
| RPO | Recovery point objective | How much data can be lost? |
| MTTR | Mean time to repair | How long does repair usually take? |
| MTBF | Mean time between failures | How reliable is it over time? |
| SLA | Service-level agreement | What service level was promised? |

## Backup types

- Full backups copy the selected data set and are simplest to restore from.
- Incremental backups copy changes since the last backup of any type and save space/time, but restore chains matter.
- Differential backups copy changes since the last full backup and grow until the next full backup.
- Snapshots capture a point-in-time state but are not always a replacement for independent backups.

## Recovery sites

- A cold site has space and basic infrastructure but little ready equipment.
- A warm site has some systems prepared and can recover faster.
- A hot site is ready to take over quickly and costs more.
- Cloud recovery can provide flexible capacity, but connectivity and data replication must be planned.

## Testing and documentation

- An untested backup is a hope, not a recovery plan.
- DR tests reveal missing dependencies, expired credentials, capacity gaps, and documentation errors.
- Runbooks should be clear enough to use under stress.

## Scenario walkthrough

1. A database can lose at most 15 minutes of data and must return within one hour.
2. The RPO is 15 minutes and the RTO is one hour.
3. Backup and replication choices must meet both numbers, not just one.

## Exam cues

- RTO is time to restore; RPO is acceptable data loss.
- Hot is fastest and usually most expensive; cold is slowest and cheapest.
- Backups must be tested. The exam loves that for a reason.
