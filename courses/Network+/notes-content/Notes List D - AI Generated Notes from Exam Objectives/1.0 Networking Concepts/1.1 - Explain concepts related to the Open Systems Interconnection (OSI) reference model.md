# 1.1 - OSI Reference Model

> **Big idea:** The OSI model is a mental map for understanding network communication. It separates the trip from an application on one system to an application on another system into seven layers, each with its own responsibility. Real protocols do not always fit perfectly into one layer, but the model is extremely useful for troubleshooting because it lets you ask, "What kind of problem is this?" before guessing at a fix.

## What you should be able to explain

- Name the seven OSI layers in order.
- Explain what each layer is responsible for.
- Connect common devices, protocols, and failures to the correct layer.
- Use encapsulation and decapsulation to reason through traffic flow.

## Concept map

| Layer | Name | Main job | Common examples |
| --- | --- | --- | --- |
| 7 | Application | Network services used by applications | HTTP, HTTPS, DNS, SMTP, DHCP |
| 6 | Presentation | Format, translate, encrypt, compress data | TLS concepts, encoding, compression |
| 5 | Session | Start, maintain, and close dialogs | Session setup, checkpoints, reconnection logic |
| 4 | Transport | End-to-end delivery between processes | TCP, UDP, ports, segmentation |
| 3 | Network | Logical addressing and routing between networks | IPv4, IPv6, routers, routing tables |
| 2 | Data Link | Local delivery on the same link | Ethernet frames, MAC addresses, switches, VLANs |
| 1 | Physical | Signals and media | Copper, fiber, radio, connectors, light levels |

## Encapsulation in plain English

- When data leaves a sender, each layer adds information that helps the next part of the network do its job.
- Layer 4 adds transport information such as TCP or UDP ports.
- Layer 3 adds source and destination IP addresses.
- Layer 2 adds source and destination MAC addresses for the current local link.
- Layer 1 turns the whole thing into electrical, optical, or radio signaling.

## Troubleshooting with the layers

- Start low when the symptom is basic: no link light, unplugged cable, bad optics, weak wireless signal.
- Move to Layer 2 when the link exists but local delivery fails: VLAN mismatch, bad trunk, duplicate MAC, switch port issue.
- Move to Layer 3 when addressing or routing looks wrong: incorrect gateway, subnet mask mismatch, missing route.
- Move to Layer 4 when the host is reachable but a service port fails: blocked TCP port, closed listener, UDP timeout.
- Move to Layer 7 when the network path works but the application behavior is wrong: DNS name, authentication, service config, or application error.

## Protocol data units

- Layers 7 through 5 are usually described as data.
- Layer 4 uses segments for TCP and datagrams for UDP.
- Layer 3 uses packets.
- Layer 2 uses frames.
- Layer 1 sends bits.

## Scenario walkthrough

1. A user can ping a server IP but cannot open the website by name.
2. Layer 1 through Layer 3 are probably functional because the IP address responds.
3. Layer 4 might be functional if the web port responds, but name-based access adds DNS.
4. The next good test is name resolution, because this smells like an Application layer DNS issue rather than a cable or routing issue.

## Exam cues

- Know the layer order cold: Please Do Not Throw Sausage Pizza Away, or any mnemonic you actually remember.
- Switches are usually Layer 2, routers are usually Layer 3, and firewalls can operate at multiple layers.
- Do not treat the OSI model as a perfect protocol taxonomy. Treat it as a troubleshooting framework.
