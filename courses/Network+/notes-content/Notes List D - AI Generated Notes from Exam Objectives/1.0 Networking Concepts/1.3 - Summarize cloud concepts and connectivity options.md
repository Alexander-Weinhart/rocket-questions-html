# 1.3 - Cloud Concepts and Connectivity Options

> **Big idea:** Cloud networking moves familiar network ideas into provider-managed environments. You still care about routing, segmentation, access control, availability, and monitoring, but the implementation uses cloud constructs such as virtual networks, subnets, security groups, gateways, load balancers, and private connectivity.

## What you should be able to explain

- Compare IaaS, PaaS, SaaS, and serverless responsibilities.
- Understand public, private, hybrid, and multicloud models.
- Explain cloud networking pieces such as virtual networks, subnets, gateways, and security controls.
- Choose connectivity options such as VPN, dedicated circuits, and peering.

## Concept map

| Model | Customer manages more of | Provider manages more of | Example use |
| --- | --- | --- | --- |
| IaaS | Operating systems, apps, data, many network settings | Physical hosts and data center | Virtual machines and virtual networks |
| PaaS | Application code and data | Runtime, OS, scaling platform | Managed database or app platform |
| SaaS | User settings and data | Almost everything else | Hosted email or CRM |
| Serverless | Functions and triggers | Runtime scaling and infrastructure | Event-driven automation |

## Virtual networks

- A cloud virtual network is a logical network boundary inside a provider environment.
- Subnets divide that virtual network into smaller IP ranges.
- Route tables control where traffic goes next.
- Security groups or network ACLs control what traffic is allowed.

## Connectivity choices

- Site-to-site VPN is flexible and usually quick to deploy, but it depends on internet path quality.
- Dedicated private connectivity can provide more predictable performance and stronger separation, but costs more and takes longer to provision.
- VPC/VNet peering connects cloud networks directly, often inside the same provider ecosystem.
- Transit gateways or cloud routers simplify many-to-many connectivity.

## Shared responsibility

- The provider secures the underlying cloud platform.
- The customer still configures access, data protection, identity, network policy, and application security.
- A cloud outage and a customer misconfiguration are different risks and need different controls.

## Scenario walkthrough

1. A business keeps an internal database on premises but runs web servers in a cloud virtual network.
2. That is a hybrid cloud design.
3. The likely connectivity choices are site-to-site VPN or a dedicated private circuit.
4. Security groups, route tables, and DNS must all agree, or the application will look broken even if the tunnel is up.

## Exam cues

- Cloud does not remove networking; it abstracts the hardware.
- Hybrid means cloud plus on-premises. Multicloud means more than one cloud provider.
- A private subnet is not magically safe. Routing and security policy decide exposure.
