# 5.1 - Troubleshooting Methodology

> **Big idea:** Troubleshooting is a structured way to move from symptom to evidence to fix. A method keeps you from guessing wildly, changing too many things at once, or declaring victory before the original problem is verified.

## What you should be able to explain

- Use a repeatable troubleshooting process.
- Gather information and define the problem clearly.
- Test theories and implement a plan safely.
- Verify full functionality and document the outcome.

## Concept map

| Step | Purpose | Practical action |
| --- | --- | --- |
| Identify the problem | Understand symptoms and scope | Ask users, check alerts, reproduce |
| Establish a theory | Choose likely causes | Map symptoms to layers and recent changes |
| Test the theory | Gather proof | Run targeted commands or checks |
| Plan and implement | Fix with risk awareness | Use change control and rollback when needed |
| Verify functionality | Confirm service is restored | Test from affected user/path |
| Document | Preserve learning | Record cause, fix, and prevention |

## Scope matters

- One user failing is different from one VLAN failing.
- One application failing is different from all network access failing.
- A recent change is evidence, not proof.
- Define the blast radius before touching configurations.

## Change one thing at a time

- Multiple simultaneous changes make it hard to know what fixed or worsened the issue.
- Take notes as you test.
- If a test disproves a theory, update the theory instead of forcing the evidence to fit.

## Verification

- The fix is not done when a command succeeds.
- Verify the original workflow from the affected location when possible.
- Check monitoring/logs after the fix to ensure the problem stays resolved.

## Scenario walkthrough

1. A user says "the network is down," but only one website fails.
2. The real scope may be DNS, a firewall rule, the remote service, or a browser issue.
3. Start by reproducing, testing other sites, checking name resolution, and comparing another user or device.

## Exam cues

- The methodology order matters.
- Always verify full system functionality after implementing a fix.
- Documentation is not optional in the exam worldview.
