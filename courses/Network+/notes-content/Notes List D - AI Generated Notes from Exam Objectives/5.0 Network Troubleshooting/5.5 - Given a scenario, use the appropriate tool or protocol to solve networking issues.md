# 5.5 - Troubleshooting Tools and Protocols

> **Big idea:** The right tool depends on the question you are trying to answer. Some tools test reachability, some reveal paths, some query names, some inspect local state, some scan services, and some prove what is actually inside packets.

## What you should be able to explain

- Match tools to symptoms and OSI layers.
- Use command-line tools for reachability, name resolution, routing, and local state.
- Understand packet capture, protocol analyzers, cable tools, and Wi-Fi analyzers.
- Choose the least disruptive tool that answers the question.

## Concept map

| Tool | Answers this question | Layer focus |
| --- | --- | --- |
| ping | Is the target reachable by ICMP? | 3 |
| traceroute/tracert | What Layer 3 path is taken? | 3 |
| nslookup/dig | What does DNS return? | 7 |
| ipconfig/ifconfig/ip | What are local interface settings? | 1-3 |
| arp | What IP-to-MAC mappings exist locally? | 2/3 |
| netstat/ss | What ports and sessions exist locally? | 4 |
| tcpdump/Wireshark | What packets are actually present? | 2-7 |
| nmap | What hosts or services appear reachable? | 3/4/7 |
| Cable tester | Is the cable wired and functional? | 1 |
| Wi-Fi analyzer | What is happening in RF space? | 1/2 |

## Command-line tools

- Use ping for basic reachability, but remember ICMP may be blocked.
- Use traceroute or tracert to see where a path appears to stop or change.
- Use nslookup or dig when the symptom involves names.
- Use local interface commands to verify address, mask, gateway, DNS, and link state.

## Packet and service tools

- Packet capture is powerful when you need proof of requests, replies, retransmissions, resets, or missing traffic.
- Nmap can identify reachable hosts and open services, but it should be used only where authorized.
- Netstat or ss helps determine whether a local service is listening and what connections exist.

## Hardware and field tools

- A toner helps locate cable runs.
- A cable tester validates wiring and can reveal opens, shorts, and split pairs.
- A visual fault locator helps find fiber breaks or bad bends.
- A Wi-Fi analyzer reveals channel use, signal, and interference.

## Scenario walkthrough

1. A user can reach a server by IP but not by hostname.
2. The best first tool is nslookup or dig because the symptom points to DNS.
3. A cable tester would not answer the question, and replacing hardware would be guesswork.

## Exam cues

- Match the tool to the symptom.
- Packet capture is evidence-rich but not always the first step.
- Use authorized scanning only; the exam may frame nmap as a controlled troubleshooting tool.
